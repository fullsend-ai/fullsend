# Implementation Plan: Phase 3 — Lock Files

## Context

Phases 1-2 of ADR-0038 are fully shipped. Phase 1 (8 PRs) added URL detection, SSRF-hardened fetching, content-addressed caching, audit logging, schema extensions, resource resolution, and CLI integration. Phase 2 (3 PRs) added skill frontmatter parsing, recursive transitive dependency resolution with cycle/depth/breadth limits, and relative URL resolution.

Phase 3 adds **lock files** — `.fullsend/lock.yaml` files that pin all resolved remote dependencies (URLs and SHA256 hashes) for reproducible harness execution. Lock files let teams commit a known-good resolution state so every environment uses the same pinned dependencies without re-resolving from the network.

Design details are in `docs/plans/universal-harness-access.md` (section "Lock file format" under "Resolved design questions") and the lock file schema in the ADR at `docs/ADRs/0038-universal-harness-access.md`.

> **Scope note:** Phase 3 adds generation and consumption of lock files. It does not add transitive dependency recording in lock entries — that is handled by the existing Phase 2 resolver, whose `Dependency` output feeds the lock file generator. Lock files record whatever the resolver produces, including transitive deps.

## PR Dependency Graph

```
PR 1 (lock package) ──> PR 2 (lock CLI + lock-aware run)
```

PRs are strictly sequential. PR 1 is a standalone package with no callers. PR 2 wires it into the CLI.

## PR 1: `internal/lock/` package

**Scope:** New standalone package for reading and writing lock files. Pure data structures and file I/O. No callers. Zero risk to existing behavior.

### Create `internal/lock/lock.go`

```go
package lock

// LockFile is the top-level structure of .fullsend/lock.yaml.
type LockFile struct {
    Version     int                    `yaml:"version"`
    GeneratedAt time.Time              `yaml:"generated_at"`
    Harnesses   map[string]HarnessLock `yaml:"harnesses"`
}

// HarnessLock records resolved dependencies for one harness.
type HarnessLock struct {
    Source       string            `yaml:"source"`
    SHA256       string            `yaml:"sha256"`
    ResolvedAt   time.Time         `yaml:"resolved_at"`
    Dependencies []DependencyEntry `yaml:"dependencies"`
}

// DependencyEntry records a single resolved remote resource.
type DependencyEntry struct {
    Field          string            `yaml:"field"`
    URL            string            `yaml:"url"`
    SHA256         string            `yaml:"sha256"`
    FetchedAt      time.Time         `yaml:"fetched_at"`
    TransitiveDeps []DependencyEntry `yaml:"transitive_deps,omitempty"`
}
```

Functions:

- `Load(path string) (*LockFile, error)` — reads and unmarshals lock file. Returns `nil, nil` if file does not exist. Rejects unsupported version numbers.
- `Save(path string, lf *LockFile) error` — atomic write (temp file + `os.Rename` + `fsync`). Creates parent directories. Defaults `Version` to 1 if zero. Prepends `# Generated by fullsend lock — DO NOT EDIT` header. File permissions 0o644 (intended for version control, unlike cache files at 0o600).
- `(*LockFile) Lookup(harnessName string) *HarnessLock` — returns snapshot of entry or nil. Nil-safe.
- `(*HarnessLock) IsStale(sourceHash string) bool` — true if harness file hash differs from stored hash. Nil-safe (returns true).
- `(*HarnessLock) LookupDep(url string) *DependencyEntry` — depth-first search through dependencies and transitive deps. Bounded by `maxLookupDepth` (32) to guard against cyclic YAML anchors.
- `(*LockFile) SetHarness(name string, hl HarnessLock)` — add or replace entry. Nil-safe (no-op).
- `(*LockFile) HarnessNames() []string` — sorted list of harness names. Nil-safe.

### Create `internal/lock/lock_test.go`

Tests (24 total):
- **Load:** valid file, file not found (returns nil), malformed YAML, unsupported version, empty harnesses, multiple harnesses, transitive deps in YAML.
- **Save:** round-trip (save + load), defaults version, header comment, atomic write (overwrite), creates nested directories.
- **Lookup:** found, not found, nil receiver.
- **IsStale:** matching hash, different hash, nil receiver.
- **LookupDep:** direct match, transitive match (nested 3 levels), not found, nil receiver, max depth guard.
- **SetHarness:** new entry, replace existing, nil receiver (no panic).
- **HarnessNames:** sorted, empty, nil receiver.

**Depends on:** Nothing

**After merge:** Lock file package available. No callers. All existing tests pass.

## PR 2: `fullsend lock` CLI + lock-aware `fullsend run`

**Scope:** Adds the `fullsend lock <agent-name>` subcommand and integrates lock file consumption into `fullsend run`.

### Create `internal/cli/lock.go`

CLI subcommand:

```
fullsend lock <agent-name> --fullsend-dir <dir> [--update] [--offline] [--max-depth N] [--max-resources N]
```

- `--fullsend-dir` (required): base directory containing the `.fullsend` layout.
- `--update`: force re-resolution even when the lock entry is current.
- `--offline`, `--max-depth`, `--max-resources`: passed through to `resolve.ResolveOpts`.

`runLock` function flow:
1. Load and validate the harness from `<dir>/harness/<agent-name>.yaml`.
2. Call `h.ResolveRelativeTo(absFullsendDir)` and `h.HasURLReferences()` — early return if no URLs.
3. Load org config and validate `AllowedRemoteResources`.
4. Compute SHA256 of the harness file for staleness detection.
5. Load existing lock file from `<dir>/lock.yaml`. If entry exists and is current (hash matches) and `--update` is not set, print "up to date" and return.
6. Call `resolve.ResolveHarness(ctx, h, opts)` to resolve all remote dependencies.
7. Build `lock.HarnessLock` from resolved `[]resolve.Dependency`, including `Field` mapping.
8. Call `lf.SetHarness(agentName, harnessLock)` and `lock.Save(lockPath, lf)`.
9. Print summary of locked dependencies.

### Create `internal/cli/lock_test.go`

Tests (8 total):
- **Generates lock file:** Harness with URL agent + URL skill → lock file created with correct entries.
- **No URL references:** Local-only harness → no lock file created.
- **Already up to date:** Lock twice without `--update` → second is a no-op.
- **`--update` forces re-resolve:** Lock, then lock with `--update` → `ResolvedAt` advances.
- **`resolveFromLock` success:** Agent in cache → harness field replaced with cache path.
- **`resolveFromLock` cache miss:** Dep pinned but not in cache → error with "not in cache" and helpful message.
- **`resolveFromLock` skill slots:** Multiple URL skills → each skill slot replaced.
- **`resolveFromLock` no partial mutation:** First dep in cache, second not → harness unchanged (atomic rollback).
- **`resolveFromLock` transitive deps:** Transitive skill deps appended to `h.Skills`.
- **`resolveFromLock` diamond dependency:** Same URL as transitive and direct → direct URL filtered out.
- **`resolveFromLock` transitive policy skipped:** Policy deps not appended to skills.

### Add `resolveFromLock` helper in `internal/cli/lock.go`

Resolves harness dependencies from a lock file entry instead of the network. For each pinned dependency:
1. Verify content exists in local cache via `fetch.CacheGet`.
2. Collect mutations (field → cache path) without applying them.
3. Only after all deps are confirmed present, apply mutations to the harness.
4. Filter out remaining URL entries from `h.Skills` (handles diamond dedup).

This two-phase approach prevents partial mutations — if any dep is missing from cache, the harness is unchanged and the caller falls back to network resolution.

### Modify `internal/cli/run.go` — lock-aware resolution

In `runAgent`, between `ValidateAllowedRemoteResources` and the existing `resolve.ResolveHarness` call, add lock file lookup:

1. Load lock file from `<fullsendDir>/lock.yaml`.
2. If lock entry exists for this harness and is not stale (harness hash matches): call `resolveFromLock(h, entry, workspaceRoot, printer)`.
3. On success: use lock deps, skip network resolution.
4. On failure (cache miss): warn, fall back to normal `resolve.ResolveHarness`.
5. If lock entry is stale: warn with message to re-run `fullsend lock`.

### Add `Field` to `resolve.Dependency`

The `Dependency` struct gains a `Field string` attribute so lock entries can map each dep back to the harness field it populates (`"agent"`, `"policy"`, `"skills[0]"`, etc.).

### Fix harness validation for URL agents

`Validate()` skips agent name validation when `IsURL(h.Agent)` returns true, since URL agents are replaced with cache paths before reaching the shell.

### Modify `internal/cli/root.go`

Register `newLockCmd()` as a subcommand.

### Modify `internal/resolve/resolve.go`

Set `dep.Field = field` in the `resolveURL` return path so the field name propagates into lock entries.

**Depends on:** PR 1

**After merge:** `fullsend lock <agent-name>` generates lock files. `fullsend run` uses lock files when present and current, with fallback to network resolution.

1. **Unit tests:** `make go-test` — all new and existing tests pass.
2. **Lint:** `make lint` passes.
3. **Local-only harness (regression):** Run an existing harness with only local paths — no lock file interaction, no behavioral change.
