# Experiment Agent v3.0 Release Notes

**Release Date:** April 15, 2026  
**Major Feature:** Discovery Mode with Review + Approve UX

---

## What's New

### 🔍 Discovery Mode (Major New Feature)

**The Problem:** Users don't know what experiments to run. Blank canvas is a barrier.

**The Solution:** Agent analyzes your codebase/docs and suggests high-value experiments.

**How it works:**
1. User chooses "Suggest experiments" mode
2. Agent reads GitHub repos + strategy docs
3. Agent identifies potential experiments (pain points, hypotheses, strategic opportunities)
4. Agent evaluates each (impact, risk, feasibility, ROI, strategic alignment)
5. Agent presents top 5 with justifications and target outcomes
6. User picks one
7. **Agent pre-populates entire experiment canvas**
8. User reviews + approves or edits
9. User fills missing info (team, timeline)
10. Final review before saving

**Value:**
- Lowers discovery barrier (no blank canvas)
- Data-driven prioritization (not gut-feel)
- Saves 10-15 min setup time (pre-population)
- Surfaces experiments users might miss

---

### ✅ Review + Approve UX Pattern (Critical UX Improvement)

**The Problem:** When agent pre-populates content, user might disagree or want changes.

**The Solution:** Explicit review + approval cycle at TWO gates.

**Gate 1: Pre-Populated Content**
```
Agent: [shows pre-populated hypothesis, metrics, costs, benefits]
       "Does this look correct?
        A) Approve
        B) Edit
        C) Start over"

User: "B - change time target from 30% to 20%"

Agent: [updates canvas, shows changes]
       "Ready to approve or edit more?"

User: "A - Approve"

Agent: "Great! Now I need team and timeline..."
```

**Gate 2: Complete Design**
```
Agent: [shows COMPLETE experiment design]
       "Does everything look correct?
        A) Save it
        B) Edit something"

User: "A"

Agent: [saves to persistent storage]
```

**Why this matters:**
- User feels in control (not prescribed to)
- Can correct agent misunderstandings
- Builds trust through transparency
- Prevents saving incorrect designs

## Updated Capabilities

### Discovery Mode Analysis Sources
- GitHub repositories (README, issues, PRs, ADRs, problem docs)
- Strategy documents (roadmaps, vision docs, goals)
- Previous experiments (learn from history)
- Team discussions (meeting notes, Slack exports)

### Evaluation Dimensions
- **Impact:** High/Medium/Low (value if successful)
- **Risk:** High/Medium/Low (cost if it fails)
- **Feasibility:** Easy/Medium/Hard (resources needed)
- **Strategic Alignment:** ⭐⭐⭐⭐⭐ to ⭐ (fits stated goals)
- **ROI Potential:** Dollar estimate if scaled
- **Timeline:** Days/Weeks/Months to results

### Experiment Presentation Format
```
### #1: Experiment Name
**Priority:** HIGH/MEDIUM/LOW
**Justification & Value Analysis:** [Why this matters, context from docs]
**Impact/Risk/Feasibility:** [Quick assessment]
**ROI Potential:** [Dollar estimate]
**Target Outcome(s):**
- Specific measurable outcome 1
- Specific measurable outcome 2
```

## Architecture: Two Modes

```
Experiment Agent v3.0
│
├─ Discovery Mode (NEW)
│  │
│  ├─ Gather context (repos, docs)
│  ├─ Analyze (identify experiments)
│  ├─ Evaluate (impact, risk, feasibility, ROI)
│  ├─ Present top 5 (with justifications)
│  ├─ User picks one
│  ├─ Pre-populate canvas
│  ├─ Review + Approve cycle ◄── NEW UX PATTERN
│  ├─ Ask for missing info
│  ├─ Final review + Save
│  └─ → Execution Mode (tracking)
│
└─ Execution Mode (v2.5-Lite)
   │
   ├─ Design experiment
   ├─ Track observations
   ├─ Generate reports
   └─ Archive
```

Both modes share:
- Bias-corrected methodology
- Persistent storage
- Devil's advocate mode
- Balanced reporting (benefits + costs)

## When to Use Each Mode

### Use Discovery Mode When:
- ❓ Don't know what experiment to run
- ❓ Have strategy docs but no validation plan
- ❓ Need data-driven prioritization
- ❓ Want agent to suggest high-value opportunities
- ❓ New to experimentation, need guidance

### Use Execution Mode When:
- ✅ Already know what to test
- ✅ Someone proposed a specific change
- ✅ Following up on previous finding
- ✅ Have hypothesis ready to validate
- ✅ Want full control over design

- ✅ Prioritization aligns with your strategic goals

- ✅ User can edit pre-populated content

- ✅ User approves before moving to missing info

- ✅ **User feels in control, not prescribed to**

---

## Known Limitations

### Discovery Mode Requires Good Context
- If docs don't mention pain points or goals, agent has little to work with
- Suggest general best-practice experiments as fallback

### Analysis Takes 2-3 Minutes
- Reading repos and docs isn't instant
- Set expectation upfront

### Pre-Population May Need Edits
- Agent makes inferences, might get details wrong
- Review + Approve cycle catches this
